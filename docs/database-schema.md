# Database Schema: Office Supply Request System

> **Status:** Draft  
> **Author:** Indigo Collier  
> **Date:** 04/20/2026  
> **Database:** Firebase Firestore (NoSQL, document-based)

---

## Overview

> Describe in 1–2 sentences how the data is organized and why. What are the top-level collections?

Data is organized into two top-level Firestore collections: `users` and `requests`. The `users` collection stores each authenticated user's profile and role, which determines what they can access in the app. The `requests` collection stores every supply request submitted by an employee, including all status changes made by a manager. The two collections are linked by the employee's Firebase Auth UID — the `requests` collection stores the `employeeId` field, which maps back to the corresponding document in `users`.

---

## Collection: `users`

> Stores user profile and role data. Created when a user first authenticates.

| Field         | Type        | Required | Description                                 |
| ------------- | ----------- | -------- | ------------------------------------------- |
| `uid`         | `string`    | Yes      | Firebase Auth UID — matches the Auth record |
| `email`       | `string`    | Yes      | User's email address                        |
| `role`        | `string`    | Yes      | Either `"employee"` or `"manager"`          |
| `displayName` | `string`    | No       | User's display name                         |
| `createdAt`   | `timestamp` | Yes      | When the user record was created            |

**Document ID:** Firebase Auth UID

---

## Collection: `requests`

> Stores every supply request submitted by an employee.

| Field           | Type        | Required | Description                                                  |
| --------------- | ----------- | -------- | ------------------------------------------------------------ |
| `id`            | `string`    | Yes      | Auto-generated Firestore document ID                         |
| `employeeId`    | `string`    | Yes      | UID of the employee who submitted it                         |
| `employeeEmail` | `string`    | Yes      | Email of the submitting employee (for display)               |
| `item`          | `string`    | Yes      | Name of the requested supply item                            |
| `quantity`      | `number`    | Yes      | Number of units requested (must be > 0)                      |
| `reason`        | `string`    | Yes      | Employee's justification for the request                     |
| `status`        | `string`    | Yes      | One of: `"pending"`, `"approved"`, `"denied"`, `"fulfilled"` |
| `createdAt`     | `timestamp` | Yes      | When the request was submitted                               |
| `updatedAt`     | `timestamp` | Yes      | When the status was last changed                             |
| `denialNote`    | `string`    | No       | Manager's reason for denial (stretch goal)                   |
| `statusHistory` | `array`     | No       | Array of `{ status, changedAt }` objects (stretch goal)      |

**Document ID:** Auto-generated by Firestore

## Request Lifecycle

```
[Submitted by Employee]
        ↓
    PENDING
   ↙       ↘
APPROVED   DENIED
   ↓
FULFILLED
```

- `pending` → `approved` — Manager approves the request
- `pending` → `denied` — Manager denies the request (optional note)
- `approved` → `fulfilled` — Manager marks the request as completed/ordered
- Any other transition is not permitted

---

## Firestore Security Rules (planned)

> These rules enforce that users can only access data they are allowed to see.

- A user can only read their own record from `users`
- A manager can read all records from `users`
- An employee can create a request and read only their own requests
- A manager can read all requests and update the `status`, `updatedAt`, and `denialNote` fields
- No user can delete a request

## Notes & Design Decisions

> Record any decisions you made here and why. This is what separates a thoughtful schema from one that just exists.

- **Scope was kept tight intentionally** — the schema contains only what the application requires. No extra fields were added speculatively. If the requirements change, the schema can be extended then.
- **Employees can only read their own requests** — the `employeeId` field on every request makes this enforceable at the Firestore security rules level, not just the UI. This protects employee privacy and prevents one employee from seeing another's submissions.
- **Managers can read all requests** — the entire purpose of the manager view is to give leadership full visibility so nothing gets missed. Restricting their read access would defeat the point of the app.
- **Fulfilled is only reachable from Approved** — a manager must make a deliberate decision (approve or deny) before a request can be marked fulfilled. This enforces the business rule that fulfillment is a consequence of approval, not a shortcut around it. The lifecycle is: `Pending → Approved → Fulfilled` or `Pending → Denied`. Any other transition is invalid and will be blocked in the API layer.
